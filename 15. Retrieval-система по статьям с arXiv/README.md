# Retrieval-система по статьям arXiv

Двухступенчатая система семантического поиска научных статей по текстовому запросу: быстрый bi-encoder для отбора кандидатов + cross-encoder реранкер для точного ранжирования.

## Задача

По текстовому запросу на естественном языке найти релевантную статью arXiv среди ~98 тысяч документов. Целевая метрика — **MRR@5 ≥ 0.91** (Mean Reciprocal Rank по топ-5 выдачи). Дополнительно — профилирование компонентов системы по времени.

## Данные

- **Корпус**: 98 213 статей arXiv (`id`, `title`, `abstract`, `categories`).
- **Тест**: 1 000 запросов (развёрнутый научный вопрос → `id` верной статьи).
- Запросы англоязычные, узкоспециализированные, с LaTeX-обозначениями (`$S^{i,j}$` сохранены как значимые).
- EDA показал: почти все документы короче 512 токенов → чанкинг не нужен, каждая статья кодируется **одним вектором** (title + abstract).

## Подход

| Компонент | Выбор | Обоснование |
|---|---|---|
| Эмбеддер (bi-encoder) | `Qwen/Qwen3-Embedding-0.6B` | Сильная модель из топа MTEB; last-token pooling, instruct-промпт для запросов; ~1.2 ГБ VRAM в fp16 |
| Векторный индекс | FAISS `IndexFlatIP` + L2-нормировка | Скалярное произведение нормированных векторов = косинусная близость; точный перебор 98K векторов укладывается в миллисекунды |
| Реранкер (cross-encoder) | `Qwen/Qwen3-Reranker-0.6B` | Переранжирует топ-10 кандидатов от bi-encoder для точного финального порядка |

Реализованы три ключевых класса/функции: `ArxivRetriever` (эмбеддер + FAISS + реранкер), `mrr_at_k` (целевая метрика с юнит-тестами), `Profiler` (замер времени по компонентам).

## Результаты

| Конфигурация | MRR@5 | Латентность |
|---|---|---|
| bi-encoder (FAISS top-5) | **0.9441** | ~5 мс/запрос |
| bi-encoder top-10 + reranker | **0.9663** | ~206 мс/запрос |

- Целевой порог 0.91 достигнут **уже первой ступенью**; реранкер добавляет **+0.0222 MRR**, исправляя ранжирование у запросов с близкими соседями внутри одной научной области.
- Временной профиль предельно асимметричен: эмбеддинг запроса (~2.3 мс) и точный поиск в FAISS (~2.9 мс) практически бесплатны — приближённые индексы (IVF/HNSW) на этом масштабе не нужны. Почти всё время съедает реранкер.

## Выводы и что дальше

- Обе конфигурации проходят порог с запасом; выбор между скоростью и точностью — продуктовое решение.
- Оптимизировать стоит только реранкер (например, сократить число кандидатов с 10 до 5).
- Для узкоспециализированного поиска предпочтительнее точность, скоростью можно пренебречь.

## Как запустить

```bash
pip install torch transformers faiss-cpu numpy pandas matplotlib
jupyter notebook retrieval.ipynb
```

В ноутбуке задайте путь `DATA_DIR` к папке с `arxiv-metadata-s.json` и `test_sample.csv`. Эмбеддинги корпуса кэшируются в `.npy`, повторный запуск их не пересчитывает.
